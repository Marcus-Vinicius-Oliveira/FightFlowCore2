sua tarefa é criar uma suíte de testes automatizados End-to-End (E2E) e de integração que valide os cenários de uso mais importantes. **Objetivo:** Automatizar os testes manuais para que possamos verificar a integridade da aplicação de forma rápida e repetível. **Tecnologia de Testes:** * Utilize um framework de testes E2E moderno como **Cypress** ou **Playwright** para simular as interações do usuário no navegador. * Para testes diretos de API, se necessário, use **Jest** com **Supertest**. **Cenários de Teste a Serem Implementados:** **1. Testes Críticos de Segurança - Multi-Tenancy:** Este é o conjunto de testes mais importante. O objetivo é provar que os dados de uma academia são completamente inacessíveis por outra. * **Cenário 1.1: Isolamento de Dados na Listagem (GET)** * **Setup:** O script deve criar programaticamente duas academias (Academia A, Academia B) com seus respectivos administradores (Admin A, Admin B). * **Ação:** 1. Autenticar como Admin A. 2. Criar um novo aluno (Aluno A1) na Academia A. 3. Autenticar como Admin B. 4. Fazer uma chamada à API para listar todos os alunos. * **Validação:** A lista de alunos retornada para o Admin B DEVE estar vazia. O teste falha se o Aluno A1 for visível. * **Cenário 1.2: Prevenção de Acesso Direto (GET por ID)** * **Setup:** Usar as mesmas academias e usuários do cenário anterior. Obter o ID do Aluno A1. * **Ação:** Autenticar como Admin B e fazer uma chamada direta à API para `GET /api/users/{id_do_aluno_a1}`. * **Validação:** A API DEVE retornar um erro `403 (Forbidden)` ou `404 (Not Found)`. O teste falha se a API retornar os dados do aluno. **2. Testes de Controle de Acesso por Papel (RBAC):** O objetivo é garantir que cada papel (role) só pode executar as ações que lhe são permitidas. * **Cenário 2.1: Restrição de Ações do Professor** * **Setup:** Na Academia A, criar um Professor (Professor P1) e uma Turma (Turma T1), associando o professor à turma. * **Ação:** 1. Autenticar na API com as credenciais do Professor P1. 2. Tentar criar uma nova turma (`POST /api/turmas`). 3. Tentar editar os dados da Academia A (`PUT /api/academias/{id_academia_a}`). * **Validação:** Ambas as chamadas à API DEVEM falhar, retornando um erro `403 (Forbidden)`. **3. Testes de Fluxo End-to-End (Funcionalidade Principal):** O objetivo é simular o fluxo de uso completo do administrador para garantir que a jornada principal está funcionando sem interrupções. * **Cenário 3.1: Jornada Completa do Administrador** * **Setup:** Nenhum. O teste deve começar do zero. * **Ação:** O script de teste deve executar a seguinte sequência de ponta a ponta: 1. Registrar uma nova "Academia C" através da página de signup. 2. Fazer login com o novo usuário administrador (Admin C). 3. Navegar até a área de gestão de usuários e criar um novo Professor (Professor P2). 4. Navegar e criar um novo Aluno (Aluno C1). 5. Navegar para a gestão de turmas e criar uma nova "Turma T2", associando o Professor P2 a ela. 6. Adicionar um horário semanal para a Turma T2. * **Validação:** Cada passo da sequência deve ser concluído com sucesso. O teste deve verificar se, ao final, o aluno, o professor e a turma existem no banco de dados e estão corretamente associados à Academia C. **Entregável:** * Integre o framework de testes escolhido ao projeto. * Crie os arquivos de teste contendo os cenários descritos acima. * Adicione um novo script ao `package.json`, chamado `"test:e2e"`, que execute toda a suíte de testes de validação com um único comando (`npm run test:e2e`). * O resultado da execução dos testes deve ser claro e indicar o sucesso ou a falha de cada cenário.
