O núcleo administrativo da aplicação está estável e validado. O próximo grande objetivo é construir o **Portal do Aluno**, criando uma experiência de valor para os membros da academia. Sua tarefa é implementar a interface e os endpoints de backend necessários para que um aluno possa fazer login, visualizar informações relevantes e acompanhar seu progresso. **Objetivo:** Desenvolver um portal seguro e funcional onde cada aluno possa acessar exclusivamente os seus próprios dados. **1. Fluxo de Autenticação do Aluno:** * **Página de Login Dedicada:** Crie uma nova página em `/portal/login` (ou similar) para o login dos alunos. * **Lógica de Primeiro Acesso:** Lembre-se que os alunos são criados pelos administradores com senhas automáticas. O fluxo de login deve lidar com isso. * **Sugestão:** Na primeira vez que um aluno faz login com a senha temporária, ele deve ser imediatamente redirecionado para uma página onde é forçado a definir uma nova senha pessoal e segura. * **Backend API:** * O endpoint de login (`POST /auth/login`) pode ser reutilizado, mas a lógica do frontend deve redirecionar o usuário para o `/portal/dashboard` se a `role` no JWT for `'ALUNO'`. **2. Funcionalidades e Páginas do Portal do Aluno:** Crie as seguintes páginas, todas protegidas, exigindo que o usuário esteja autenticado com a `role` de `ALUNO`. * **Página Principal (`/portal/dashboard`):** * **Conteúdo:** Deve exibir uma mensagem de boas-vindas ("Olá, [Nome do Aluno]!"). * **UI/UX:** Apresentar um resumo rápido e útil, como um card mostrando "Sua Próxima Aula" (com base na grade de horários da sua turma) e talvez um contador "Total de Presenças este Mês". * **Página de Horários (`/portal/horarios`):** * **Funcionalidade:** Exibir a grade semanal completa de aulas da academia. * **UI/UX:** Deve ser uma versão **somente leitura** do componente de calendário visual que o administrador utiliza. O aluno pode ver todas as aulas, mas não pode interagir para editar ou marcar presença. Destaque visualmente as aulas das turmas em que o aluno está matriculado. * **Página de Frequência (`/portal/minha-frequencia`):** * **Funcionalidade:** Mostrar o histórico de presença pessoal do aluno. * **UI/UX:** Apresente os dados de forma clara, talvez com uma lista cronológica de aulas e o status (`Presente`, `Falta`). Um filtro por mês seria uma excelente melhoria. **3. Requisitos de Backend (Novos Endpoints de API):** Crie os seguintes endpoints na API NestJS, todos protegidos pelo `AuthGuard` e pela lógica de `role`. * `GET /api/portal/meus-dados`: Retorna os dados do aluno logado (nome, faixa, etc.), extraindo o `userId` do JWT. * `GET /api/portal/minha-frequencia`: Retorna o histórico de presença **apenas do aluno logado**. A query no banco de dados deve ser `WHERE aluno_id = :userIdDoJWT`. * `GET /api/portal/minhas-aulas`: Retorna a lista de próximas aulas para as turmas em que o aluno logado está matriculado. **4. Requisitos de Segurança (Não Funcionais):** * **Escopo de Dados:** É absolutamente crítico que cada um desses novos endpoints garanta que um aluno só possa acessar seus próprios dados. Em nenhum momento um aluno (`Aluno A`) deve ser capaz de consultar a frequência ou os dados de outro aluno (`Aluno B`). * **Consistência de UI:** Mantenha o estilo visual (cores, fontes, componentes) do Portal do Aluno consistente com o restante da aplicação já desenvolvida. Execute este plano para construir o Portal do Aluno, completando um ciclo de valor fundamental para o nosso SaaS.
